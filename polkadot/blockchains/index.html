Polkadot (DOT) Carbon Footprint Methodology

The Genesis block of the Polkadot network was launched on May 26, 2020. On August 21, 2020 DOT was revalued. From this date onwards, one DOT (old) equals 100 new DOT. We ignore any historical data before the revaluation date.
Messari data starts from 03 Sep 2020, and this is the first date for which we calculate DOT’s carbon footprint.

Electricity consumption

Node power demand

The Polkadot wiki has a reference hardware spec:
  • CPU
    • x86-64 compatible;
    • Intel Ice Lake, or newer (Xeon or Core series); AMD Zen3, or newer (EPYC or Ryzen);
    • 4 physical cores @ 3.4GHz;
    • Simultaneous multithreading disabled (Hyper-Threading on Intel, SMT on AMD);
    • Prefer single-threaded performance over higher cores count. A comparison of single-threaded performance can be found here.
  • Storage
    • An NVMe SSD of 1 TB (As it should be reasonably sized to deal with blockchain growth). An estimation of current chain snapshot sizes can be found here. In general, the latency is more important than the throughput.
  • Memory
    • 16GB DDR4 ECC.
  • System
    • Linux Kernel 5.16 or newer.
  • Network
    • The minimum symmetric networking speed is set to 500 Mbit/s (= 62.5 MB/s). This is required to support a large number of parachains and allow for proper congestion control in busy network situations.
We then use the outervision power supply calculator to estimate the power demand using the following configuration:
  • Server motherboard
  • Intel Core i7-975
  • 16GB DDR4 RAM
  • NVMe SSD storage
  • Always on (24/7)
The estimated power demand is 258W per node.

Network emissions

We use the total number of active nodes (not all nodes) to calculate the total electricity consumption of the network.
Realtime active node count is available from the subscan validator endpoint.

Node distribution

The geographic distribution of Polkadot nodes is provided by the Web3 foundation at their locationstats API endpoint.
This endpoint gives the location stats of selected nodes participating in Web3 Foundation’s Polkadot Thousand Validators Programme which covers more than 50% of all active validators.
The staking system pays out rewards essentially equally to all validators regardless of staked amount.
Therefore we can simply estimate the distribution of computational power using the distribution of nodes by number.
From the locationstats endpoint we can see at least 30% of nodes run on cloud infrastructure. Nodes running on cloud infrastructure are in general 70% more energy efficient.
We make a conservative assumption that 25% of nodes are in datacentres.
The locations of the nodes are accurate to city level. For cities in the USA, Canada and Australia, we use the state or province of the city as the region because we have state-level emission factors for grid electricity. For other locations, we use country-level or global emission factors.

Transaction count and fees

In Polkadot, Extrinsics can take two forms, “inherents” and “transactions”.
For simplicity, we take the number of extrinsics per day as a proxy for the number of transactions. The transaction count is not essential here for accounting purposes but we do need total transaction fees.
We use Subscan as the source for transaction information.

Staking rewards

Reliable and accessible historical data for Polkadot is not available. We scan through all events generated by the staking module and sum the balance for all events of type “Rewarded” and “Reward” on a given date to calculate the daily total staking rewards.
