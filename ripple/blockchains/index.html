Ripple Carbon Footprint Methodology

  • The XRP Ledger (XRPL) is the open-source distributed ledger created by Ripple. The native cryptocurrency of the XRP Ledger is XRP.
  • Compared to Bitcoin, which uses a distributed blockchain whose transactions are processed and secured by proof-of-work (PoW) mining, XRP transactions are processed by a network of trusted validators on the XRPL.
  • XRP does not rely on Proof of Work (PoW) and therefore there is no mining involved with XRP
  • XRP instead relies on a consensus algorithm known as the Ripple Protocol Consensus Algorithm. The XRPL’s integrity is maintained by a group of trusted nodes. All transactions must be agreed by a super-majority of these trusted nodes for the network to achieve consensus and for the transactions to be included in the XRPL.
  • Anyone can become a validator. However, Ripple maintains a given set of validators that can be trusted. This trusted list of nodes is called the Unique Node List.
We treat the XRPL like a proof of stake (PoS) network for the purposes of calculating its carbon footprint, since the emissions from the network come from nodes processing and validating transactions in a way which is computationally similar to PoS networks.

Ripple Block Rewards

There are no staking rewards from being a validator on the Ripple network (it is not a PoS protocol but transactions are still validated using a consensus mechanism between nodes).
The main reason to run a validator is to maintain and preserve the stable performance and progression of the network. It is the validators who determine the development of the XRP Ledger, so any company that employs or depends on the XRPL has an incentive to guarantee the security of the network.
Accordingly, in our carbon footprint calculation, all emissions from the XRPL are allocated to transactions, with no carbon footprint from merely holding XRP.

Electricity Consumption

Validator Power Requirements

Ripple provides the minimum and recommended hardware requirements for running a validator. Ripple specify that the minimum requirements should only be used for testing purposes or occasional use. So we use the recommended requirements to calculate the electricity consumption of validators.
The recommended requirements specify the use of an Intel Xeon processor. Using this, we found the minimum spec CPU which satisfies the recommended requirements (more than 3 GHz frequency with a minimum of 8 cores) to be the Intel Xeon E5-2637.
Ripple recommends disk speed over 10,000 reads/s and 7,000 writes/second. We assume a more efficient NVMe SSD for our typical validator node, rather than a SATA SSD.
From the above the outervision power calculator gives us a power consumption 224W. We use this figure, rather than assuming a more efficient cloud VM[^1], in order to be conservative with our estimate.

Nodes vs Validators

  • XRPL is a peer-to-peer network of independent nodes which process transactions and maintain the shared state of the ledger
  • Nodes can be run by anyone
  • Nodes can be one of the following types:
    • Stock Node (API Server): Stock nodes elect validators
    • Validator Node: Validate transactions and their sequence
    • Hub Node: Relay information to everyone
All nodes play a role in processing transactions, so we include all node types in our emissions calculations.

Validator Distribution and Number

Realtime geographic distribution of validators is taken from the livenet node distribution. This provides IP addresses for nodes, which we use to get their locations. This is also the source used for official XRPL published data.
Historic data on node distribution is not available, so we extrapolate from the current geographic distribution to calculate historical emissions figures. Network activity in terms of throughput, transaction count, and transaction fees has been consistent from 2020-01-01 to the present, and the geographic distribution of the network has not changed significantly since we started collecting data, so extrapolating in this way seems reasonable.

Blockchain Metrics

All historical metrics (e.g. transaction count, transaction fees, supply) are taken from the Messari API.

Assumptions And Limitations

  • This methodology does not yet include embedded carbon emissions from the manufacture and disposal of the computers used as XRPL validator nodes.

Comparison

The official Ripple carbon calculator gives 0.0079 kWh/tx as the energy consumption per transaction, and a figure for their annual emissions of 270 tCO2e, or 739.72 kgCO2e per day.
By contrast, our calculations show daily emissions of 1,476.58 kgCO2e, reflecting our more conservative estimate of the power requirements of XRPL nodes.
Our emissions per transactions is 0.001324 kgCO2/tx whereas Ripple’s carbon calculator estimates it as approximately 0.0045 kgC02/tx. It is interesting that our emissions per transaction is lower yet daily emissions are higher. This suggests that Ripple are calculating emissions using a lower figure for transaction count. This seems plausible, since their analysis uses 2019 data.
Ripple also publish their own sustainability report. Our weighted daily emission factor is lower than Ripple’s - 378.321873 gCO2e/kWh whereas Ripple’s is 570 gCO2e/kWh. This is probably because our validator distribution includes all node types (validators, stock nodes, hub nodes) and many of these may be in countries with lower emission factors. Also, Ripple’s analysis is from 2019, and several countries now have lower emissions factors for grid electricity, from efforts to transition to renewable sources.
[^1]: Ripple recommend the use of an AWS EC2 “z1d.2xlarge” VM for running a validator.
